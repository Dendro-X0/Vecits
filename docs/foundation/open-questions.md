# Open Questions

These questions are intentionally unresolved. They define the exploration frontier of the project.

## Value and incentives

- Should credits represent reward, access, reputation-linked entitlement, or protocol-limited purchasing power?
- Who is allowed to create credits, and under what evidence requirements?
- Should provider rewards always be newly minted, or should some exchanges destroy buyer credits without direct transfer?
- How should refunds work without turning the system into normal transferable money?
- How should the system reward valuable maintenance or rescue work that does not have strong immediate buyer demand?

## Trust and identity

- What level of pseudonymity is acceptable?
- How should social recovery or identity rotation work?
- How much should trust depend on human vouches versus delivery history?
- What is the minimum viable sybil resistance for an experimental first release?

## Marketplace design

- Which service types are objective enough for no-human-arbitration settlement?
- How should vague or partially subjective work be excluded or constrained?
- Should offers be open-ended listings, negotiated contracts, or standardized templates first?
- How small should milestones be to keep disputes tolerable?

## Disputes and protocol outcomes

Current locked operating model: [market-operating-model.md](market-operating-model.md). Open research below extends but does not override D9 (no kernel human arbitration).

- What evidence should count as sufficient delivery proof?
- When both parties disagree, what protocol loss profile is fair enough to discourage abuse?
- Should deadlock always penalize both sides?
- Are there any narrow cases where third-party attestations are acceptable without becoming governance?

## Reputation

- Should reputation be global or category-specific?
- Should reputation decay over time?
- How much should repeated activity with the same counterparty count compared with diverse successful exchanges?
- How should dispute history affect visibility and matching?

## Scope and strategy

- Is the most valuable first use case freelance work, open-source maintenance, AI services, or something else?
- Does the project need a marketplace first, or a contribution and reward network first?
- What would make this system meaningfully better than existing freelance platforms for one narrow user group?
- What kinds of failure would still be acceptable in an experimental alpha?
- Should v0 center on a mutual-aid lane for stalled technical projects before expanding into broader service exchange?
